# 命理宝鉴 · 持续推进机制（顶层架构）

> **目标**：你不需要主动催命理宝jian。它会自己推进、自己日报、自己告警。
> **生效日**：2026-07-23
> **核心思想**：4 角色 + 3 步闭环 + 1 总指挥

---

## 1. 顶层架构：4 个固定角色

```
┌─────────────────────────────────────────────────┐
│  你（决策者）：只在 ①任务变更 ②阻塞项 ③失败 ≥3 次 时介入  │
└──────────────────┬──────────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────────┐
│  总指挥（main session / preset_2）                  │
│  · 接收心跳和 cron 任务                            │
│  · 调度 worker 执行                              │
│  · 汇总产出、回报你                                │
└──────────────────┬──────────────────────────────┘
                   │
        ┌──────────┼──────────┬──────────┐
        ▼          ▼          ▼          ▼
    ┌───────┐ ┌────────┐ ┌───────┐ ┌────────┐
    │ Worker│ │ Health │ │ Kanban│ │ Cron   │
    │ 开发者 │ │ Checker│ │ Manager│ │ Runners│
    └───┬───┘ └───┬────┘ └───┬───┘ └────┬───┘
        │         │          │          │
        └─────────┴─────┬────┴──────────┘
                        ▼
              ┌──────────────────┐
              │ 共享状态层       │
              │ · KANBAN.md     │  ← 任务队列
              │ · health.log    │  ← 健康历史
              │ · workspace/    │  ← 产出物
              └──────────────────┘
```

### 4 角色职责

| 角色 | 谁 | 何时工作 | 做什么 |
|------|-----|---------|--------|
| **总指挥** | main session（preset_2） | 每次心跳/cron 唤醒 | 读 KANBAN → 派单 → 汇总 |
| **Worker 开发者** | session_spawn 子 agent | 每个推进节点 | 写代码/文档/测试 |
| **Health Checker** | health-check.sh | 每次心跳 | 端口/路由/响应体检 |
| **Kanban Manager** | KANBAN.md 维护器 | 每次任务节点完成 | 节点进度 X/8 推进 |
| **Cron Runners** | 9 个定时任务 | 各自 cron 触发 | 推送/监控/数据/报周 |

## 2. 3 步闭环：每次心跳的标准流程

```
   心跳触发
     │
     ▼
① 诊断 ──► 跑 health-check.sh
            · 全 OK → 继续
            · 有 ISSUES → 派 Worker 修复 → 重新跑
     │
     ▼
② 推进 ──► 读 KANBAN.md「进行中」
            · 找当前任务和「下一步动作」
            · 派 Worker 执行（最多 30 分钟）
            · Worker 完成后更新 KANBAN
     │
     ▼
③ 汇报 ──► 输出 ≤200 字进度卡片
            · 通过 message 推送给你（webchat）
            · 含本次产出路径 + 下一步
```

**核心约束**：
- ① 健康检查失败 ≥3 次 → **升级阻塞**，告警你决策
- ② Worker 超时（>30 分钟）→ kill + 重派 + 计入重试
- ③ 没产出 ≠ 没推进 → 必须有文件写入或明确登记"延后原因"

## 3. 1 个总指挥：main session 的责任清单

main session 每次唤醒必须按顺序回答 5 问：

| # | 问题 | 失败动作 |
|---|------|---------|
| 1 | 5 个服务都活着吗？ | 派 Worker 重启/排查 |
| 2 | 当前进行中任务是哪个？节点几/几？ | 派 Worker 执行下一步 |
| 3 | 这一步的产出物在哪？ | 没产出 → 返工 |
| 4 | 阻塞项 ≥2 天没解决？ | 升级告警 |
| 5 | 距上次推进多久？≥3 次空推进？ | 强制重启某任务 |

## 4. 失败兜底：3 道闸

### 闸 1：空推进检测
- 每次心跳必须产生 ≥1 个文件写入 或 KANBAN 节点数 +1
- 否则记一次"空推进"，累计 3 次 → 任务标记 ⚠️ 并告警

### 闸 2：超时熔断
- Worker 单次执行 ≤30 分钟
- 超过 → kill、重派，第 2 次失败 → 跳过任务并告警

### 闸 3：总指挥失声检测
- 距上次心跳成功汇报 ≥2 小时 → 启动「健康上报任务」强制 main session 汇报
- 启动方式：cron `every 2h` 触发一个独立的"状态上报"任务

---

## 5. 你（决策者）的介入时机

只在以下 3 种情况需要你：

1. **任务变更** —— 加新任务、改优先级、跳过任务
   - 指令：`插队 #X：任务描述` 或 `跳过 #Y 拉 #Z`
2. **阻塞项决策** —— KANBAN 标红的项
   - 看 KANBAN.md「当前阻塞项」表
3. **≥3 次失败** —— 同一任务连续 3 次推进失败
   - 会被自动告警推送

**其他时间你可以**：不发指令。系统会自己推进、汇报。

## 6. 手动指令速查表

| 你说的 | 系统做什么 |
|--------|----------|
| `推进 #X` | 派 Worker 立即执行 #X 的下一步 |
| `看 KANBAN` | 输出当前进度快照（≤200 字） |
| `插队 #X：描述` | 新任务插到 P0 顶部 |
| `跳过 #Y` | 任务 Y 移到 P2 末尾 |
| `加速` | 心跳间隔 30 分钟 → 10 分钟（临时） |
| `减速` | 心跳间隔 30 分钟 → 60 分钟 |
| `暂停` | 关闭所有心跳推进（保留健康检查） |
| `恢复` | 重新打开心跳推进 |
