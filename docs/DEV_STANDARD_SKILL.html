二、11 项开发规范

P0-1 API 设计规范

URL /api/v{1,2,...}/ 前缀 + 复数名词 + kebab-case

  • 必须用 apiResp() 统一响应壳
  • 动作走子路径：/yuanzhus/:id/push-yearly
  • 旧路径保留 6 个月 + Deprecation 响应头

P0-2 前端组件库规范

强制使用受控 HTML 组件，禁止裸写

  • rich-card / rich-grid / rich-panel
  • rich-table / rich-badge / rich-dot
  • rich-timeline / rich-metrics / rich-compare

P0-3 错误处理与降级规范

三级错误码 + 前端 ERROR_HANDLERS 强制实现

  • 400xxx 参数 / 401xxx 鉴权 / 5xxxxx 服务
  • 401001 → 跳登录 / 429001 → 30s 倒计时
  • 503001 → KB 兜底 / 5xxxxx → 一键重试

P1-4 性能基线与预算规范

5 项性能指标 + Lighthouse CI + bundle 检查

  • API P95 ≤ 500ms / H5 首屏 ≤ 3s
  • JS bundle ≤ 1.2MB / CSS ≤ 300KB
  • 图片 ≤ 1MB/张

P1-5 可观测性规范

5 类 25 指标 + 5 告警规则

  • HTTP / 引擎 / DB / KB / AI 五维指标
  • SLOW_REQUEST / HTTP_5XX / ENGINE_ERROR
  • 告警去重 1 分钟内同 key 不重复

P1-6 测试规范

金字塔 4 层测试

  • 单测 ≥ 80% / 集成 ≥ 60%
  • E2E：9 大流程 26/26 已通过
  • 压测：autocannon P95 ≤ 500ms

P1-7 国际化与文案规范

所有用户可见中文字符串迁 i18n 字典

  • app/i18n/zh-CN.json + en-US.json
  • 错误文案从 ERROR_COPYWRITING.md 抄

P2-8 可访问性 a11y 规范

WCAG 2.1 AA

  • 颜色对比度 ≥ 4.5:1
  • 键盘可导航 + aria-label 完整

P2-9 隐私与合规规范

PII 加密 + GDPR 式接口

  • 手机号 SHA256 + 后 4 位明文
  • 身份证 AES-256 / 地址模糊到区/县
  • 用户匿名化 + 账户删除 + 审计日志

P2-10 变更与发布规范

分支策略 + 版本号 + CHANGELOG

  • main 受保护 / gh-pages 部署
  • 版本号 vYYYY.MM.DD-patch
  • CHANGELOG 按版本号分章节

P2-11 文档即代码规范

每次 P0 完成 → 更新 docs/index.html + CHANGELOG

  • 架构 / API / 数据 / 部署 / 报告 / 索引
  • 强制按四要素起草：现状 + 要点 + 验收 + 节奏

四、开发流程铁律

多轮迭代：一轮完成自动进入下一轮
任务范围：拆成 5-10 分钟小任务，避免超时
交付前自检：JS / HTML / API / DB schema 全检
路径铁律：绝对路径 + 人话链接 + 端口 8920/8930
KB 学习：必须分 8 轮提取，单轮正则不够
git commit：符合 <type>(<scope>): <subject> 格式

五、检查清单（开发前/后必看）

5.1 开发前

确认任务属于哪个 T 层级（T1-T4）
确认目标端（H5 / 小程序 / 微信 H5 / PC 后台 / 智能眼镜）
阅读相关规范文档（API_STANDARD / UI_COMPONENT / ERROR_HANDLING）
检查是否复用既有组件（rich-card / grid / timeline 等）

5.2 开发后

JS 语法 node --check 通过
HTML 标签闭合，HTMLParser 验证通过
API 走统一响应壳 apiResp()
错误码从 ERROR_CODES 常量引用
受控 HTML 组件使用正确（rich-* 类名）
文件路径绝对路径 + 人话链接文字

5.3 提交前

Git commit 信息符合格式
双分支同步：main + gh-pages
CHANGELOG 更新
关键报告归档到 docs/
