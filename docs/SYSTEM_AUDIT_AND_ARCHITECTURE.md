## 一、系统审计摘要

共发现 **21个安全问题**，按严重程度分布：

### P0 严重风险（4项）— 需立即修复

| # | 问题 | 文件位置 | 影响 |

| P0-1 | API密钥硬编码在前端JS | ai-interpreter.js:37、divination-core.js:2985/3137/15956、calc-engine-lib.js:2822 | 攻击者可从浏览器提取g2claw.com的Bearer Token，冒用产生高额费用 |

## 二、双平台权限架构

### 2.1 平台关系

```
┌─────────────────────────────────────────────────────┐
│                 命理宝鉴 · 统一网关                   │
│                (API Gateway + JWT Auth)              │
├────────────────────────┬────────────────────────────┤
│  平台A · 国学体系       │  平台B · 周易中医诊疗        │
│  (会员权限驱动)         │  (角色权限驱动)             │
│  排盘/学习/商城/课程    │  病患→大师→医生→报告推送    │
├────────────────────────┴────────────────────────────┤
│           共享层 · 知识库 (60 JS文件 / 13.2MB)        │
│      按角色分级：公开/会员/大师专用/病患安全           │
├─────────────────────────────────────────────────────┤
│           数据层 · PostgreSQL + Redis                 │
│    平台A数据 │ 平台B数据 │ 共享KB索引 │ 用户身份      │
└─────────────────────────────────────────────────────┘
```

| 免费会员 | free | 注册 | 基础排盘、部分KB |

| 平台A管理员 | admin_a | 分配 | 会员/商城/KB管理 |

**平台B · 周易中医诊疗（4角色制）**

| 角色 | 编码 | 获取方式 | 核心权限 |
|------|------|----------|----------|
| 病患 | patient | 实名注册 | 提交症状、查看养生报告 |

| 角色 | 编码 | 说明 |
|------|------|------|
| 超级管理员 | super_admin | 隐含A5+B4+S1全部权限 |

**角色组合规则：** 一个账号可同时拥有多角色（如 `["vip", "patient"]`），JWT的roles字段为数组，后端取并集校验。

### 2.4 周易术语隔离机制


## 四、大师诊断结论单独存放

### 4.1 案例库数据模型

```
master_cases (大师案例库)
├── 案例ID、UUID、编号
├── 关联：master_id, patient_id, related_case_id(复诊链)
├── 状态：draft → submitted → reviewing → approved → completed → archived
├── 八字排盘(加密) + 五行摘要(明文，用于统计)
├── 症状(加密) + 体质(明文)
├── 大师分析(加密) + 分析摘要(JSON，用于检索)
├── 医学化翻译(加密)
├── 医生审核：诊断(加密) + 审核状态 + 审核意见
├── 最终方案(加密)
├── 质量标记：质量分(1-10) + 高质量标记 + 疗效评级(1-5)
└── 时间戳：创建/提交/完成/归档

master_case_versions (版本历史)
├── 每次修改存为一个版本
├── 版本类型：master_analysis / doctor_diagnosis / final_plan
└── 内容加密 + SHA-256哈希校验

case_discussions (讨论记录，加密)
case_review_logs (审核流程记录)
patient_pseudo_mapping (脱敏ID映射，最高安全级别)
```

## 五、知识蒸馏提取机制

### 5.1 五步蒸馏流程

```
Step 1: 案例收集
  ├── 触发：每月/每季度 / 新增高质量案例≥50条 / 检测到未覆盖新模式
  ├── 筛选：status=completed + quality_score≥7 + effectiveness≥3
  └── 输出：候选案例集

Step 2: 模式提取
  ├── 特征提取：五行格局/脏腑风险/卦象/体质/证候/方剂/疗效
  ├── 模式聚合：按(五行+体质+证候)分组聚类，统计有效率
  ├── 置信度计算：confidence = (支持案例数 × 平均有效率) / 总案例数
  └── 阈值：confidence≥0.6，支持案例≥5，矛盾比例≤20%

Step 3: 知识验证
  ├── 自动验证：与现有KB交叉比对 + 逻辑一致性 + 方剂-证候匹配 + 去重(>85%)
  ├── 人工审核：大师审周易部分 + 医生审中医部分 + 管理员审分类
  └── 三方均通过才入库

Step 4: KB更新
  ├── 匹配目标KB文件（白名单7个文件可接受蒸馏）
  ├── 写入策略：新增/补充/修正，每条标注来源案例ID
  └── 安全检查：无个人信息泄露 + 格式一致 + 生成diff

Step 5: 版本管理
  ├── 版本号：MAJOR.MINOR.PATCH (MINOR级递增)
  ├── 快照存储 + diff记录
  ├── 管理员确认后生效
  └── 支持回滚到任意历史版本
```

### 5.2 蒸馏白名单

| KB文件 | 可接受蒸馏类型 |
|--------|----------------|
| bazi-knowledge-base.js | 格局-证候关联 |
| tcm-diagnosis-kb.js | 体质-证候-方剂模式 |
| tcm-famous-formulas-kb.js | 方剂加减规律 |
| zhouyi-knowledge-base.js | 卦象-病机关联 |
| nihaisha-tcm-kb.js | 养生调理方案 |
| authoritative-knowledge-base.js | 综合规律 |
| wuxing-correspondence.js | 五行-脏腑关联 |

### 5.3 知识来源追踪

每条蒸馏写入的知识标注：
```javascript
// [DISTILL] batch=2026-07-000 cases=[CASE-001,CASE-007,CASE-012,...]
//            confidence=0.78 effectiveness=0.83 verified_by=master#42,doctor#17
//            written_at=2026-07-19T11:30:00Z
```

可通过 `distillation_knowledge` 表反查：这条知识来自哪些案例 → 这个案例贡献了哪些知识 → 这次蒸馏批次写了什么 → 这个KB文件的版本历史。
