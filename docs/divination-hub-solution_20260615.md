## 第一章：总体需求规格说明书（PRD级）

### 1.1 项目背景与愿景

#### 1.1.1 行业背景

中华传统文化绵延五千年，命理学、风水学、中医养生、儒释道信仰构成了庞大的知识体系。然而，这些智慧长期面临三大困境：

1. **知识碎片化**：散布于典籍、民间口传、各派师承之中，缺乏系统性整理。一个想学习八字的普通人，需要在《渊海子平》《三命通会》《滴天髓》《子平真诠》等多部古籍之间反复对照，入门门槛极高。
2. **传承断层化**：老一辈命理师逐渐凋零，年轻一代对传统文化接触有限，口诀、经文、仪轨面临失传风险。
3. **工具孤岛化**：现有命理App多聚焦单一功能（如只做八字或只做风水），用户需在多个应用间切换，缺乏一站式解决方案。且大多数App知识浅薄，仅有排盘功能而无专业解读。

#### 1.1.2 项目愿景

**"乾元命理宝鉴"致力于成为全网最权威、知识体系最丰富、最通俗易懂的命理智能体与信众修行平台。**

具体愿景拆解：

- **知识维度**：建立10大命理领域 + 3大信仰体系的完整知识图谱，覆盖从入门到精通的学习路径
- **工具维度**：提供八字、六爻、风水、姓名学、测字、中医体质、紫微斗数、奇门遁甲、梅花易数、六壬等全系列数术工具
- **修行维度**：为信众提供经文诵读、口诀持修、参拜指导、养生药方等日常修行辅助
- **AI维度**：借助大语言模型（LLM）实现千人千面的命盘解读、AI解卦、智能取名、风水咨询
- **文化维度**：以数字化手段保存和传播中华古老智慧，让传统文化在数字时代焕发生机

#### 1.1.3 核心价值主张

| 价值 | 说明 |
|------|------|
| **一站式** | 学习、查询、占卜、修行，无需跳转其他平台 |
| **权威性** | 知识来源于经典古籍，由专业命理框架保障算法正确性 |
| **通俗性** | 每条知识配备白话解读，零基础用户也能理解 |
| **个性化** | 基于用户出生信息提供定制化的运势、吉日、修行建议 |
| **活性化** | 知识库持续增长，AI自动补充，用户反馈驱动优化 |

### 1.2 核心用户群体画像

#### 1.2.1 用户群体一：普通信众

| 属性 | 描述 |
|------|------|
| **典型画像** | 30-55岁，有一定传统文化认同，日常拜佛/拜神/看黄历，但不精通风水命理术语 |
| **核心诉求** | 今日宜忌、本命吉日、求签问卜、经文诵读、养生保健 |
| **使用频率** | 每日1-3次（晨起看黄历、睡前诵经） |
| **痛点** | 网上信息杂乱、真假难辨；不同来源说法矛盾；不懂专业术语 |
| **功能偏好** | 黄历 → 每日运势 → 经文伴读 → 参拜指导 → 吉日查询 |
| **转化路径** | 免费基础功能 → 深度报告（付费） → 功德打赏 |

#### 1.2.2 用户群体二：命理爱好者

| 属性 | 描述 |
|------|------|
| **典型画像** | 20-45岁，对八字/六爻/风水等有基本认知，希望系统性学习或验证自己的判断 |
| **核心诉求** | 精准排盘、专业解读、知识库查询、学习路径、不同流派对比 |
| **使用频率** | 每周3-7次，每次15-45分钟 |
| **痛点** | 现有App排盘不准/知识太浅；缺乏体系化学习资料；无法对比不同流派观点 |
| **功能偏好** | 八字排盘 → 六爻占卜 → 知识库深度学习 → 风水分析 → 导出报告 |
| **转化路径** | 免费排盘 → 高级解读（AI分析） → 知识库会员 → 线下课推荐 |

#### 1.2.3 用户群体三：修行者

| 属性 | 描述 |
|------|------|
| **典型画像** | 35-65岁，已皈依或有固定修行习惯（诵经/打坐/站桩），对佛道儒某一体系有深入信仰 |
| **核心诉求** | 经文全文（带拼音）、口诀持修、修行日历、功德记录、体质养生 |
| **使用频率** | 每日2-5次（早课/晚课/持咒/查看吉日） |
| **痛点** | 经文拼音标注难找；不同版本经文有出入；持修缺乏打卡和记录工具 |
| **功能偏好** | 经文伴读 → 口诀持修 → 功德签到 → 养生药方 → 参拜指导 |
| **转化路径** | 免费经文 → 功德系统 → 修行社群 → 法会信息 |

#### 1.2.4 用户群体四：取名/咨询客户

| 属性 | 描述 |
|------|------|
| **典型画像** | 25-40岁，新生儿取名/公司起名/改名的刚需用户，或需要命理咨询的决策者 |
| **核心诉求** | 五格五行分析、八字配合取名、AI智能推荐、导出专业报告 |
| **使用频率** | 低频（按需使用），但ARPU高 |
| **痛点** | 取名软件只看五格不看八字；名字寓意解读太机械；缺乏综合判断 |
| **功能偏好** | 姓名学 → 公司取名 → 八字排盘（配合） → AI分析 → 报告导出 |

### 1.4 非功能性需求

#### 1.4.1 性能需求

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 首屏加载时间 | ≤ 3秒（本地部署）/ ≤ 5秒（远程） | 关键路径JS不超过500KB |
| 八字排盘响应 | ≤ 500ms | 纯前端计算，无网络依赖 |
| 六爻掷卦响应 | ≤ 200ms | 随机数+查表 |
| 知识库搜索响应 | ≤ 1秒 | SQLite FTS5全文检索 |
| AI分析响应 | ≤ 30秒 | 依赖LLM API，需用户等待提示 |
| 并发用户 | ≥ 50（本地部署） | Python HTTP Server默认线程池 |
| JS文件总大小 | ≤ 2MB（gzip后≤500KB） | 分模块懒加载 |

#### 1.4.2 兼容性需求

| 平台 | 最低版本 | 说明 |
|------|----------|------|
| Chrome | 90+ | 主要目标浏览器 |
| Safari | 14+ | iOS用户主力 |
| Firefox | 88+ | 次要支持 |
| Edge | 90+ | Chromium内核，与Chrome同 |
| iOS Safari | 14+ | 移动端必须支持 |
| Android Chrome | 90+ | 移动端必须支持 |
| 屏幕尺寸 | 320px - 2560px | 响应式全覆盖 |
| Web Speech API | 仅Chrome/Safari完整支持 | 语音伴读需降级处理 |

#### 1.4.3 安全性需求

| 需求 | 实施方案 |
|------|----------|
| 用户数据本地化 | 所有用户数据（八字/六爻记录/功德）存储在localStorage/SQLite，不上传云服务器 |
| API通信加密 | LLM API调用通过本地代理（8900端口），浏览器到代理为localhost通信 |
| Token保护 | Gateway Token仅在api-proxy-server.py中使用，不暴露给前端 |
| XSS防护 | 所有用户输入经过转义后再渲染；经文/口诀数据为预定义常量，不接受用户输入 |
| CORS限制 | API代理仅允许localhost来源 |
| 数据完整性 | SQLite启用WAL模式，定期VACUUM |

#### 1.4.4 国际化需求（简繁英）

| 语言 | 优先级 | 覆盖范围 | 说明 |
|------|--------|----------|------|
| 简体中文 | P0 | 100% | 主语言，所有功能完整覆盖 |
| 繁体中文 | P1 | 80% | 经文保持原文（多为繁体），UI需翻译；港台信众市场 |
| English | P2 | 50% | 数术工具UI + 核心知识库；海外华人及对东方文化感兴趣的西方用户 |

国际化实施方案：

1. **数据层**：知识库条目增加 `i18n` 字段，含 `zh-CN`/`zh-TW`/`en` 三种语言的内容
2. **UI层**：使用 `data-i18n` 属性标注可翻译元素，通过JS动态切换
3. **术语层**：建立命理术语翻译对照表（如"八字"→"Four Pillars of Destiny"、"十神"→"Ten Gods"）
4. **经文层**：佛经保留梵文+中文原文，增加英文简释；道经保留原文+英文注释

### 2.6 数据增长策略

#### 2.6.1 增长路线

| 阶段 | 时间 | 数据量 | 增长方式 |
|------|------|--------|---------|
| 当前 | 2026-06 | ~470KB（JS文件合计） | 人工编写 + AI辅助 |
| Phase 1 | 2026-Q3 | ~1.5MB | AI批量生成 + 人工校验 |
| Phase 2 | 2026-Q4 | ~3MB | 自动爬取经典文本 + 结构化 |
| Phase 3 | 2027-Q1 | ~5MB | 社区贡献 + AI持续填充 |
| Phase 4 | 2027-Q2+ | ~10MB+ | SQLite迁移 + 全文检索 + 增量更新 |

#### 2.6.2 数据增长策略详解

**策略一：AI辅助批量生成**

- 每个知识领域，使用LLM生成初稿（格式化JSON），人工校验后入库
- 单次对话可生成1-2个领域的完整条目（约20-50KB）
- 校验重点：术语准确性、古籍引用正确性、白话解读通俗性

**策略二：经典文本结构化**

- 佛经/道经/儒典原文可从CBETA/道藏/十三经等权威数字典籍获取
- 结构化处理：分段 → 拼音标注 → 白话翻译 → 注释 → 关联
- 拼音标注使用pypinyin库自动生成 + 人工校对多音字

**策略三：JS→SQLite迁移**

- 当前JS文件超过500KB时，性能开始下降
- 迁移至SQLite：数据存储在DB中，前端通过API按需加载
- 优势：全文检索、增量更新、数据完整性、备份便利

**策略四：社区贡献机制**

- 用户可提交知识条目（经人工审核后入库）
- 修行者可贡献经文读音、口诀持修心得
- 命理师可贡献实战案例（脱敏处理后）


#### 3.1.2 核心算法/逻辑说明

1. **年柱推算**：以立春为界，立春前为上一年。年干 = (年份-3) % 10，年支 = (年份-3) % 12
2. **月柱推算**：以节令为界（非中气），月干由年干推算（五虎遁年起月法），月支固定
3. **日柱推算**：使用儒略日数法推算日干支，基准日为1900年1月1日（甲戌日）
4. **时柱推算**：时干由日干推算（五鼠遁日起时法），时支对应十二时辰
5. **十神配置**：以日干为我，按五行生克+阴阳同异确定十神
6. **神煞推算**：基于天干地支组合查表，含天乙贵人、文昌、驿马、桃花、华盖、羊刃、空亡等
7. **格局判断**：以月令透干为格，结合四柱组合判断正格/杂格
8. **用神选取**：扶抑法（旺则泄耗、弱则生扶）+ 调候法（寒暖燥湿）+ 通关法（两行相战）+ 从格法
9. **大运排列**：阳男阴女顺排、阴男阳女逆排，以月柱为基准
10. **流年推演**：逐年天干地支与命盘生克关系分析

#### 6.5.3 懒加载策略

```javascript
// 模块懒加载器
const ModuleLoader = {
  loaded: {},
  
  async load(moduleName) {
    if (this.loaded[moduleName]) return;
    
    const moduleMap = {
      bazi: ['js/bazi.js', 'data/authoritative-knowledge-base.js'],
      liuyao: ['js/liuyao.js', 'data/hexagram-data.js'],
      fengshui: ['js/fengshui.js'],
      xingming: ['js/xingming.js'],
      cezi: ['js/cezi.js', 'data/char-database.js'],
      zhongyi: ['js/zhongyi.js'],
      faith: ['js/faith.js', 'data/faith-knowledge-base.js', 'data/scripture-database.js', 'data/koujue-database-full.js'],
      knowledge: ['js/knowledge.js', 'data/authoritative-knowledge-base.js'],
      export: ['js/export.js'],
    };
    
    const files = moduleMap[moduleName] || [];
    for (const file of files) {
      await this.loadScript(file);
    }
    this.loaded[moduleName] = true;
  },
  
  loadScript(src) {
    return new Promise((resolve, reject) => {
      const script = document.createElement('script');
      script.src = src;
      script.onload = resolve;
      script.onerror = reject;
      document.head.appendChild(script);
    });
```

### 6.6 知识库自动填充机制

#### 6.6.1 AI驱动填充流程

```
1. 检测缺失 → 知识库覆盖率检查（每周）
2. 生成Prompt → 基于目标结构生成LLM提示词
3. AI生成 → LLM生成JSON格式知识条目
4. 人工校验 → 命理术语准确性、古籍引用正确性
5. 入库 → 写入JS/SQLite，更新版本号
6. 验证 → 前端加载验证，确认显示正确
```

#### 6.6.2 自动填充Prompt模板

```
你是中华命理学的权威专家。请为「{领域}」的「{主题}」生成完整的知识条目。

要求：
1. 严格遵循以下JSON结构：{结构模板}
2. 每个字段必须填写完整，不得留空
3. 引用的古籍必须真实存在，引用内容必须准确
4. 白话解读要通俗易懂，避免堆砌术语
5. 五行、天干、地支等基础概念必须与传统命理学一致

参考典籍：{经典来源列表}
当前已有条目示例：{已有条目}

请生成{数量}个完整条目。
```

#### 6.6.3 渐进填充计划

| 会话 | 填充内容 | 预计增量 |
|------|---------|---------|
| 每次对话 | 1-2个知识模块 | 5-20KB |
| 每周优化 | 补充缺失领域 | 20-50KB |
| 专项会话 | 整个领域深度填充 | 50-200KB |

#### 6.6.4 质量控制

1. **术语准确性**：AI生成的命理术语需与《渊海子平》《三命通会》等经典对照
2. **古籍引用验证**：引用的古籍名、篇名、原文必须可查证
3. **白话通俗性**：非专业用户能否理解
4. **数据一致性**：同一概念在不同领域的描述是否一致
5. **交叉引用**：新增条目需与已有条目建立交叉引用
