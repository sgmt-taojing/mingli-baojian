**版本**: 2026.06.23 v2
**定位**: 全网最专业命理智能体平台 — 实战推算 + 古法传承 + AI增强
**架构师**: AutoClaw

## 产品核心原则

1. **实战为王** — 每一种命理推算技术必须工具化，给出可信赖的推算结论和化解之法
2. **古法+AI** — 推算模型全量引用古老智慧经典，叠加AI大模型推演，结论更客观、方法更简单
3. **知识全量** — 知识类内容全量收集，形成真正的知识库，可供平台所有入口引用
4. **自进化** — 知识库可自进化，日趋完善
5. **功能完整易用** — 从实战出发，完整、易用、好用
6. **UI专业化** — 界面专业、美观、流畅

## 二、架构演进建议

### 2.1 用户思考回应

> "命理宝鉴已有微信公众号后，是否可以不需要移动端网页，保留 PC 端后台管理和移动端微信公众号？"

**结论：推荐此方案。** 以下是详细分析。

### 2.2 方案对比

| 维度 | 方案 A：PC后台 + 移动端独立网页 | 方案 B：PC后台 + 微信公众号 H5（推荐） |
|------|------|------|
| **开发成本** | 需独立开发移动端页面、域名、备案、SEO | 复用现有 H5 页面，微信内嵌即可 |
| **用户获取** | 需要独立推广，获客成本高 | 微信 13 亿用户池，公众号自然引流 |
| **用户留存** | 依赖浏览器书签，回访率低 | 公众号关注即留存，推送触达 |
| **支付能力** | 需接入第三方支付 | 微信支付原生集成（未来变现） |
| **分享传播** | 需要分享按钮+链接 | 朋友圈/群聊原生分享，转化率高 |
| **功能限制** | 无限制 | 微信内嵌浏览器有部分限制（但够用） |
| **审核风险** | 独立站点内容自控 | 微信平台有审核机制，需合规运营 |
| **维护成本** | 两套前端（PC + 移动端） | 一套 H5 适配微信即可 |

### 2.3 推荐方案：PC 端后台管理 + 微信公众号 H5

```
┌─────────────────────────────────────────────┐
│              PC 端后台管理                    │
│  (功能开发 / 内容管理 / 数据分析 / 引擎测试)  │
│  http://127.0.0.1:8901/app/divination-hub   │
└──────────────────┬──────────────────────────┘
                   │ API 服务层
                   │ (knowledge-server:8901 + api-proxy:8900)
                   ▼
┌─────────────────────────────────────────────┐
│           微信公众号 H5 服务                  │
│  (用户触达 / 内容推送 / H5 工具 / 互动)       │
│  微信内嵌浏览器 → cloudflared tunnel → 本地   │
└─────────────────────────────────────────────┘
```

### 2.4 利弊分析

**优势：**
- **降低维护成本**：不需要维护独立移动端站点，H5 页面微信适配即可
- **用户自然增长**：公众号推送 → 阅读原文 → H5 体验 → 分享 → 新关注
- **合规可控**：微信平台审核机制倒逼内容合规，降低法律风险
- **变现路径清晰**：未来可通过微信支付（会员/商城）实现商业化

**劣势与应对：**
- 微信内嵌浏览器性能略差 → 优化 H5 加载速度，2.1MB 主文件需做分包加载
- 服务号每月仅 4 次推送 → 精选内容，重质不重量；日常通过模板消息补充
- 需微信认证（300 元/年）→ 成本可控，认证后才能菜单跳转 H5
- 平台依赖风险 → 保留 PC 端独立访问能力，不把鸡蛋放一个篮子

### 2.5 过渡策略

```
当前状态                    过渡期                      目标状态
─────────                ─────────                  ─────────
PC端完整功能    →    PC端后台管理化      →    PC端纯后台
所有H5页面           H5微信适配               微信公众号H5服务
无公众号              注册+认证公众号           完整运营体系
本地访问              cloudflared隧道           正式域名+HTTPS
```

**分三步走：**

1. **第一步（1-2 周）**：PC 端保持现有功能不变，新增微信适配层（UA 检测、CSS 适配、分享卡片）
2. **第二步（2-4 周）**：注册公众号、完成认证、配置菜单、部署 cloudflared 隧道
3. **第三步（4-8 周）**：PC 端逐步转型为后台管理界面（增加内容管理、数据分析功能），移动端用户全部引导至公众号

## 十、专业补充

### 10.1 架构师视角的风险与应对

#### 风险 1：单文件过大（divination-hub.html 2.1MB）

**现状**：47 个 script 块，34906 行代码集中在一个 HTML 文件中。

**风险**：
- 加载缓慢（尤其移动端/微信环境）
- 维护困难（改动一处可能影响全局）
- 浏览器内存压力
- 无法按需加载

**建议**：
- **短期**：将内嵌 JS 提取为外部文件，按功能模块拆分（bazi.js / ziwei.js / qimen.js 等）
- **中期**：实现按需加载（用户点击某功能时才加载对应引擎）
- **长期**：考虑引入轻量构建工具（如 Vite）做模块化管理和代码分割

**优先级**：⭐⭐⭐⭐（在微信 H5 适配前完成基本拆分）

#### 风险 2：知识库数据与代码耦合

**现状**：知识库以 JS 全局变量形式存储，直接被 HTML 引用。

**建议**：
- 将知识库迁移为 JSON 格式（已有 knowledge-db.json 雏形）
- 通过 API 动态加载，而非 script 标签硬编码
- 便于后续接入 CMS 或数据库

**优先级**：⭐⭐⭐

#### 风险 3：服务端可靠性

**现状**：Python http.server 单线程，无并发处理能力。

**建议**：
- 短期可满足开发需求
- 若公众号上线后用户量增长，需迁移到 async 框架（如 FastAPI / aiohttp）
- 或使用 Gunicorn 多 worker 部署

**优先级**：⭐⭐（用户量达到 500+ 时需处理）

#### 风险 4：cloudflared 隧道稳定性

**现状**：使用 cloudflared 临时隧道暴露本地服务。

**建议**：
- 临时隧道 URL 每次重启会变化，不适合生产环境
- 需配置固定域名（cloudflared tunnel + 自定义域名）
- 或使用 ngrok / frp 等替代方案
- 微信公众号菜单跳转需固定 URL

**优先级**：⭐⭐⭐⭐（公众号上线前必须解决）

### 11.2 硬代码修复记录

| # | 问题 | 风险 | 修复 |
|---|------|------|------|
| 1 | 神仙诞辰用 Math.random() | 结果不可复现 | computeDaysUntilLunarEvent（真实农历） |
| 2 | 风水户型用 Math.random() | 结果不可复现 | 图片宽高比分析 |
| 3 | 流年月度12月硬编码 | 不符合命理逻辑 | computeMonthLuck（五虎遁+日主五行） |
| 4 | 2处 while(true) 无退出 | 可能死循环 | 加365安全计数器 |
| 5 | 黄历tips随机 | 每次刷新不同 | 按日期轮转 |
| 6 | 五维评分随机 | 每次刷新不同 | 按日期轮转 |
| 7 | 名言tips随机 | 每次刷新不同 | 按日期轮转 |
| 8 | 梅花起卦随机 | 用户无感知 | 加toast提示 |

**核心教训**：命理系统中的随机数是毒药——所有结果必须可复现、有逻辑依据。日期轮转是替代随机的简单有效方案。

### 11.3 架构演进经验

1. **纯前端架构的极限**：2.1MB 单文件已接近可维护极限，必须拆分
2. **知识库 JSON 化**：JS 全局变量方式不利于查询和管理，JSON + API 是正确方向
3. **微信适配要趁早**：后期适配比初期适配成本高 3 倍
4. **合规先行**：功能开发前先确认合规边界，避免返工
5. **引擎测试用例**：每次引擎修改后必须回归测试，排盘引擎的回归测试是刚需
6. **定时任务管理**：已有3个 cron 任务运行良好，建议将知识库检查、知识推送等也纳入 cron
7. **子代理并行**：知识库扩充等可并行任务，使用子代理可提速 3-4 倍
